## Executive synthesis

For an LLM harness that must support natural, low-latency, full-duplex-feeling voice interaction, the most practical architecture is usually a **chained voice pipeline** rather than an end-to-end speech-to-speech model: capture live audio, run streaming speech-to-text, feed stabilized text into the LLM, stream the LLM’s output into a text-to-speech engine, and play audio incrementally while continuously listening for interruptions. That approach preserves explicit control over transcripts, tools, policy checks, context management, fallback behavior, and model switching. OpenAI’s current voice-agent guidance makes the same distinction between direct speech-to-speech sessions and chained voice pipelines, recommending the chained design when you want stronger control over intermediate text, transcript durability, and existing text-agent reuse. citeturn17view2turn17view0

The deepest implementation challenge is **not** any single model. It is the coordination problem across six time-sensitive loops: microphone capture, VAD and turn detection, interim and final ASR updates, LLM time-to-first-token and inter-token latency, streaming TTS, and barge-in handling. LiveKit’s turn-detection and interruption docs, Pipecat’s voice pipeline docs, Deepgram’s endpointing controls, Google Cloud’s voice-activity events, and Azure’s real-time recognition and streaming synthesis docs all point to the same conclusion: the product experience depends on how well you decide **when the user is done, when to start generating, when to speak, and when to stop speaking** in response to user interruption. citeturn23view0turn23view1turn23view2turn24view0turn24view2turn14view0turn15view0turn15view2turn16view0

For the LLM layer, **vLLM is the strongest general-purpose open-source serving framework** when you need high-throughput, low-latency text generation with streaming outputs, OpenAI-compatible APIs, prefix caching, continuous batching, and distributed parallelism. TensorRT-LLM is especially compelling when you are committed to NVIDIA infrastructure and want the deepest optimization stack around in-flight batching, paged KV cache, speculative decoding, and Triton integration. `llama.cpp` is excellent for edge, CPU-heavy, or small-scale deployments, but it is less naturally positioned as the central serving layer for a multi-tenant, GPU-dense voice backend. citeturn18view2turn18view0turn18view1turn18view3turn10search2turn19view0

For ASR, **raw Whisper is not a native streaming engine**. The official repository states that `transcribe()` processes audio with a sliding 30-second window, which is fine for file transcription but not ideal for conversational turn-taking. `faster-whisper` materially improves speed and memory efficiency and is often the best self-hosted Whisper-family option, but if the requirement is highly responsive live dialog with strong endpointing and interim result semantics, purpose-built streaming or conversational ASR systems such as Deepgram Flux/Nova-3, Google Cloud STT, Azure Speech, or Amazon Transcribe generally fit the problem better. The Whisper-Streaming research line shows why: it can retrofit Whisper-like models to streaming, but the cited paper still reports about **3.3 seconds average latency**, which is respectable for live captioning yet usually too slow for premium back-and-forth voice assistants. citeturn12view0turn11view0turn28view2turn26search0turn14view1turn15view1turn26search3turn26search2

For TTS, the key requirement is **incremental output from incremental text**. Azure Speech exposes streaming synthesis and explicit first-byte and finish-latency metrics, plus text streaming over a WebSocket v2 endpoint. OpenAI’s TTS supports streaming audio output. ElevenLabs supports both chunked-transfer streaming and a WebSocket API intended for partial text input. Coqui XTTS offers open-source multilingual streaming with claimed sub-200 ms latency, while Piper is attractive for local/offline use but is less feature-rich and the original Rhasspy repository is archived, with active development having moved to the Open Home Foundation fork. citeturn16view0turn20view0turn20view1turn20view2turn22search6turn21view0turn21view1turn22search0

## Reference architecture

A browser voice client should begin with `getUserMedia()` in a secure context, which yields a `MediaStream` from the microphone. From there, the preferred transport to a media-capable edge or speech gateway is usually **WebRTC** if the client is a browser or mobile app and the server side needs real-time media semantics. MDN and the W3C position WebRTC as the browser-native real-time communication stack, and even OpenAI’s current Realtime guidance recommends WebRTC over WebSockets for browser live-audio connections because it tends to deliver more consistent performance for real-time audio. citeturn27view0turn1search9turn1search5turn17view1

A practical reference architecture looks like this:

```text
Browser or mobile client
  -> Mic capture and optional local AEC/NS
  -> WebRTC media session to edge gateway
  -> Edge voice session manager
       -> local VAD and speech activity events
       -> streaming ASR
       -> transcript stabilizer and turn manager
       -> LLM orchestrator
            -> session state and tool calls
            -> vLLM or TensorRT-LLM or hosted LLM
       -> response chunker
       -> streaming TTS
       -> outbound audio stream
  -> WebRTC audio playback to client

Cross-cutting
  -> traces, metrics, logs
  -> rate limits, admission control, retries
  -> interrupt and cancel signals
```

The cleanest internal service boundary is usually: **WebRTC at the edge, gRPC or internal event streams inside the backend**. gRPC natively supports bidirectional streaming, deadlines, cancellation, and metadata, making it a strong fit for service-to-service message flows among the ASR, orchestration, LLM, and TTS layers. WebSockets remain useful for simpler server-to-server or backend-to-browser transports, but the browser WebSocket API does not provide built-in backpressure handling, which is relevant when streaming audio chunks or rapid interim events. citeturn27view2turn27view1

For orchestration, there are two good patterns. One is a **framework-managed real-time session**, as seen in LiveKit Agents, where turn detection, interruption handling, and media transport are first-class capabilities. The other is a **pipeline-and-frame architecture**, as used by Pipecat, where each stage emits and consumes typed frames and high-priority control signals. LiveKit is particularly strong when WebRTC, telephony, or managed turn handling are core requirements. Pipecat is especially attractive when you want a Python-native orchestration layer and explicit frame-level control over STT, LLM, TTS, and interruption logic. citeturn23view0turn23view1turn24view0turn24view1

The LLM service should usually sit **behind a session manager**, not be called directly by the transport layer. That session manager owns transcript stabilization, context trimming, policy checks, tool execution, cancellation, TTS chunking, and barge-in state. With vLLM, that service can take advantage of streaming outputs, prefix caching, and—once traffic justifies the complexity—disaggregated prefill/decode to improve time-to-first-token versus inter-token latency trade-offs. citeturn18view2turn18view0turn18view1

## Latency, turn-taking, and logical breakpoints

Human conversation places unusually tight constraints on system timing. Cross-linguistic turn-taking research has found mean gaps between turns on the order of **roughly 100–200 ms**, and broader turn-taking research continues to emphasize how short those gaps are in natural dialog. Real voice systems usually cannot match that exactly because they must detect speech boundaries, decode speech, generate text, synthesize audio, and transport media. Even so, those human baselines are useful: they imply that a voice assistant that waits a full second or two after the user stops speaking will feel sluggish, even if the text answer is good. citeturn2search4turn2search8

A good engineering target is therefore an **end-of-user-speech to first-assistant-audio** time of roughly **500–800 ms** for the happy path, with even faster reaction for interruption handling. Pipecat’s own overview explicitly frames a normal round trip in that 500–800 ms band, while Azure’s synthesis docs emphasize that user-perceived latency is mainly determined by first-byte latency plus network round-trip time and that streaming playback should start as soon as the first audio chunk arrives. This target is an engineering inference, not a universal standard, but it is directionally aligned with both human turn-taking research and current voice-platform guidance. citeturn24view0turn16view0turn2search4turn2search8

The core design mistake to avoid is **equating silence with turn completion**. VAD is necessary, but VAD alone is not sufficient. WebRTC VAD works on 10, 20, or 30 ms frames and is fast and battle-tested, while Silero VAD can process a 30+ ms chunk in under 1 ms on a CPU thread and is designed to be lightweight and portable. Those tools are excellent for detecting speech presence, but LiveKit and Pipecat both stress that VAD-only systems can cut users off during thinking pauses because they do not understand semantics or prosody. That is why modern systems layer **semantic or acoustic turn detection on top of VAD**. citeturn13view1turn13view0turn23view0turn24view2

The recommended breakpoint policy is a **hybrid endpointing model**:

1. **Speech start**: trigger on VAD onset after a few consecutive voiced frames and begin buffering audio immediately.  
2. **Interim transcript phase**: consume partial ASR updates, but treat the newest words as unstable. Azure, Deepgram, and Amazon Transcribe all make clear that interim outputs may change before the utterance is finalized.  
3. **Endpoint candidate**: if silence appears, do not immediately commit the turn; instead check for a semantic end cue such as completed intent, falling intonation if available, punctuation emergence, or a dedicated turn detector score.  
4. **Turn commit**: commit when a silence threshold and an end-of-thought signal align, or when the ASR provider emits a confident endpoint event.  
5. **Preemptive generation**: optionally let the LLM start work before turn commit, but do not begin audible TTS until the system is reasonably sure the user has yielded the floor.  

This design follows the behavior exposed by Google Cloud voice activity events, Deepgram endpointing and `utterance_end_ms`, Azure’s recognizing-versus-recognized distinction, and LiveKit’s explicit treatment of turn detection, endpointing, and preemptive generation as separate controls. citeturn15view0turn14view0turn15view2turn23view1

Handling **partial user utterances** requires a transcript model with two regions: a **committed prefix** and an **editable tail**. Amazon Transcribe describes streaming transcription as partial results that settle into final results per natural speech segment, and its partial-result stabilization feature exists precisely because the tail portion of the transcript can keep changing. Azure similarly distinguishes `Recognizing` events, which are provisional, from `Recognized` events, which are final. This means the session manager should never inject the entire live interim transcript into the durable LLM context as if it were final. Instead, it should continuously update a scratch transcript and only append stable text to conversation memory at turn commit or after explicit stabilization thresholds. citeturn26search2turn26search6turn26search7turn15view2

**Raw Whisper is the wrong mental model for this problem.** The official Whisper implementation processes full audio with a sliding 30-second window, and the Whisper-Streaming paper shows that adding simultaneous streaming behavior is nontrivial and still lands around 3.3 seconds average latency. That is why a Whisper-family system is reasonable for a first self-hosted prototype, especially using `faster-whisper`, but a premium production voice experience usually benefits from either a purpose-built streaming ASR stack or a more advanced semantic turn layer on top of Whisper-family interim hypotheses. citeturn12view0turn11view0turn28view2

On the response side, the breakpoint rule should be: **chunk TTS at clause or sentence boundaries, not token boundaries**. ElevenLabs explicitly supports partial-text WebSocket input for chunked generation, Azure supports text streaming to TTS over WebSocket v2, and current streaming-TTS research shows that bounded lookahead and prosodic boundary awareness improve quality and stability. The practical implication is simple: let the LLM stream tokens, but hold a small text buffer and flush to TTS when you see punctuation, a clause boundary, or enough semantic completeness—often a short phrase, not a single token and not an entire paragraph. citeturn20view1turn16view0turn28view0turn28view1

During assistant playback, interruption handling should be **audio-first and cancellation-capable**. LiveKit’s adaptive interruption model is a useful reference point: it analyzes overlapping speech acoustically to distinguish genuine barge-ins from short acknowledgments like “uh-huh” or “okay,” instead of naively stopping every time it hears new speech. Pipecat’s defaults also keep interruptions enabled because natural conversation expects users to be able to cut in mid-sentence. In practice, your harness should support at least three interruption states: true barge-in with immediate cancel, probable backchannel with continued playback, and false interruption where playback resumes from an audio/text cursor if feasible. citeturn23view2turn24view2

For **self-hosted STT**, the strongest Whisper-family default is `faster-whisper`, not OpenAI’s raw Whisper package. The official `faster-whisper` README documents both speed and memory improvements over `openai/whisper`, supports GPU and CPU quantization, and integrates Silero VAD. Whisper itself remains attractive because of its multilingual robustness and model availability, and the official README still provides useful model-size guidance, including approximate VRAM needs from about 1 GB for tiny/base up to around 10 GB for large, with `turbo` positioned as a faster optimized variant of `large-v3`. citeturn11view0turn12view0

For **cloud STT**, the best fit depends on whether your largest problem is transcription quality or turn-taking quality. Deepgram Flux is explicitly marketed as a conversational STT model for voice agents and exposes endpointing, interim results, speech-start events, and utterance-end events. Google Cloud exposes voice activity events in real time and current streaming recognition over gRPC. Azure Speech provides real-time transcription with intermediate results and time alignment. Amazon Transcribe’s streaming docs are especially useful if you care about managing unstable transcript tails and stabilization trade-offs. citeturn26search0turn14view0turn15view0turn15view1turn26search3turn26search2turn26search6

For the **LLM layer**, vLLM is the most balanced recommendation for a custom harness because it combines streaming outputs, OpenAI-compatible and gRPC interfaces, continuous batching, prefix caching, distributed parallelism, and support across multiple hardware families. The underlying PagedAttention work exists specifically to improve memory handling for large-scale serving. TensorRT-LLM becomes more attractive when you can standardize on NVIDIA and want tighter control over in-flight batching, paged KV caching, and Triton deployment. `llama.cpp` is strongest when you want portability, quantized local models, and a lightweight HTTP server with continuous batching and speculative decoding, but it is usually the wrong first choice for a central, horizontally scaled multi-session voice backend unless your model sizes are modest or you are optimizing for edge deployment. citeturn18view2turn18view1turn5search0turn18view3turn10search2turn19view0

For **TTS**, there are three distinct categories. Azure and ElevenLabs are the best examples of mature streaming TTS platforms with explicit support for chunked audio return and, in Azure’s case, detailed first-byte telemetry and text streaming over WebSocket v2. OpenAI’s current TTS supports streamed audio output and integrates naturally if you are already using other OpenAI audio components. Coqui XTTS is the best-known open-source option when you want relatively high-quality multilingual streaming with a modern feature set. Piper is still relevant for offline or embedded-style deployments, but the original Rhasspy repository is archived; if you want that ecosystem, the current Open Home Foundation fork is the healthier reference point. citeturn16view0turn20view2turn20view1turn20view0turn22search6turn21view0turn21view1turn22search0

For **turn management**, use a layered stack rather than a single detector. WebRTC VAD is fast and simple. Silero VAD adds a stronger open-source CPU option and is often easier to integrate into Python-heavy stacks. Then add a semantic or acoustic turn detector, because both LiveKit and Pipecat explicitly document the shortcomings of VAD-only systems for conversational endpointing. If you choose a managed STT like Deepgram Flux, part of that turn intelligence can move into the ASR service itself. citeturn13view1turn13view0turn23view0turn24view2turn26search0

## Infrastructure and operations

The best default split is **thin client, smart edge, GPU-heavy backend**. The client should own microphone access, playback, echo cancellation and noise suppression where the platform already provides them, and resilient reconnection logic. The edge should own session establishment, media ingress and egress, local VAD or speech events, interruption signals, and admission control. The backend should own ASR, turn logic, LLM orchestration, TTS, policy checks, tools, and durable state. This split keeps latency-sensitive media handling close to the user while keeping the model-serving complexity centralized. citeturn27view0turn17view1turn24view0

On compute planning, the **LLM usually dominates latency and cost**, not VAD. Silero VAD is tiny and CPU-friendly, and WebRTC VAD works on tiny frames. Whisper-family self-hosting is more variable: the official Whisper README gives approximate VRAM requirements for model tiers, while `faster-whisper` publishes representative GPU and CPU benchmark comparisons showing how strongly quantization and batching affect speed and memory. That means you should treat STT sizing as a benchmarking exercise rather than a one-size-fits-all rule, but it is usually feasible to keep VAD on CPU, place ASR on CPU or a smaller GPU pool, and reserve premium GPUs for the LLM serving tier. citeturn13view0turn13view1turn12view0turn11view0

For scaling the LLM path, the most important operational distinction is **prefill versus decode**. vLLM’s disaggregated prefilling exists precisely because time-to-first-token and inter-token latency behave differently and may deserve separate optimization. TensorRT-LLM and Triton likewise emphasize in-flight batching and dynamic batching as the mechanisms that raise throughput without blindly sacrificing latency. In other words, do not scale a voice backend by simply “adding more GPU.” Scale it by separating the latency-sensitive decode path from heavy prompt prefill work, constraining queue delays, and measuring how batching affects TTFT, ITL, and tail latency. citeturn18view0turn18view3turn6search3turn10search6turn6search11

Observability has to be **end-to-end across the session**, not isolated per model. vLLM already exposes production metrics at `/metrics` and includes OpenTelemetry examples. More broadly, OpenTelemetry is designed to unify traces, metrics, and logs across distributed systems, while Prometheus remains the default metrics sink in many inference stacks. For a voice harness, the minimum useful trace spans are: audio ingress, VAD onset, ASR interim latency, ASR finalization latency, turn-commit time, LLM TTFT, LLM token streaming rate, TTS first-byte latency, playback start, interruption detection, and cancellation propagation. Without those spans, tuning the system becomes guesswork. citeturn25search1turn25search4turn25search2turn6search13turn25search3

Error handling should be designed around **fast degradation, not heroic retries**. If WebRTC fails, fall back to a WebSocket PCM path or text chat where appropriate. If interim ASR is unstable, keep listening longer rather than forcing a bad early response. If TTS generation stalls, return text in the UI immediately and either retry speech or switch to a lower-latency backup voice. If the user interrupts, cancel downstream LLM and TTS work as early as possible; gRPC supports deadlines and cancellation semantics, and modern turn-handling frameworks explicitly model interruption as a first-class control signal. citeturn27view2turn23view1turn24view1turn23view2

## Recommended implementation path

The fastest credible path is an **MVP chained pipeline** with browser capture, WebRTC transport, local VAD, streaming ASR, a single orchestrator service, vLLM for text generation, and a managed streaming TTS service. That combination keeps the core harness under your control while offloading the hardest “make it sound good immediately” problem—streaming prosodic speech—to a mature provider. If you want to stay almost entirely open source on day one, use `faster-whisper` plus Silero VAD, vLLM, and Coqui XTTS; if you want the shortest path to a polished product, use Deepgram Flux or Azure Speech for STT and Azure, ElevenLabs, or OpenAI for TTS. citeturn11view0turn13view0turn18view2turn26search0turn26search3turn20view2turn20view0turn16view0turn22search6

A strong **open-source stack** today is: LiveKit or Pipecat for voice-session orchestration; WebRTC transport; Silero or WebRTC VAD; `faster-whisper` for STT; vLLM for the LLM; and Coqui XTTS or the OHF Piper fork for TTS. This stack gives you full control and avoids vendor lock-in, but it shifts responsibility for voice quality, multilingual testing, latency tuning, and production operations squarely onto your team. It is best for teams that value transparency, on-prem deployment, or tight data control over the last increment of perceived polish. citeturn23view0turn24view0turn13view0turn13view1turn11view0turn18view2turn22search6turn22search0

A strong **production-grade stack** for most product teams is: WebRTC client transport; LiveKit-style session management or an equivalent custom session layer; managed conversational or streaming STT such as Deepgram Flux, Google Cloud STT, or Azure Speech; vLLM or TensorRT-LLM for the core text model if you want to retain control of the LLM tier; and managed streaming TTS such as Azure Speech, ElevenLabs, or OpenAI. This mix is usually the best compromise between controllability and customer-visible responsiveness, because you keep the text brain and orchestration logic while outsourcing the speech subsystems that are hardest to perfect. citeturn23view1turn26search0turn15view1turn26search3turn18view2turn18view3turn16view0turn20view2turn20view0

The self-hosted versus managed trade-off is straightforward. **Self-hosted** gives you data control, model portability, and cost leverage at scale, but it also means you own turn detection quality, speech quality, multilingual QA, GPU scheduling, and every outage. **Managed services** reduce implementation time and usually improve edge-case behavior around endpointing, prosody, and network resilience, but they narrow your control surface and may complicate cost or compliance discussions. The best compromise for many teams is hybrid: self-host the LLM harness, buy STT/TTS first, then internalize speech components later if the economics or privacy requirements warrant it. citeturn17view2turn18view2turn18view3turn26search0turn16view0turn20view1

The implementation phases should be:

**Phase one** should prove the conversational loop: microphone capture, one transport, one ASR, one LLM, one TTS, basic interruption, and end-to-end traces. Do not optimize for maximum model quality first; optimize for stable turn-taking and measurable latency. Google, Deepgram, Azure, LiveKit, and Pipecat all expose features that make this phase measurable—voice activity events, endpointing, recognizing versus finalized text, preemptive generation controls, and pipeline events. citeturn15view0turn14view0turn15view2turn23view1turn24view1

**Phase two** should improve breakpoint logic: swap VAD-only endpointing for semantic turn detection, introduce committed-prefix versus editable-tail transcript handling, enable preemptive LLM generation, and add clause-based TTS chunking with interruption-aware cancellation. That is the phase where the system begins to sound conversational rather than merely functional. citeturn23view0turn24view2turn26search2turn26search7turn28view0turn20view1

**Phase three** should harden for production: separate prefill and decode if needed, add concurrency controls, deploy metrics dashboards, benchmark GPU memory and queue delays, and introduce fallback providers for STT or TTS. vLLM’s metrics and disaggregated prefilling, Triton’s dynamic batching and model analyzer, and OpenTelemetry plus Prometheus give you the core tooling for this stage. citeturn25search1turn18view0turn6search3turn6search11turn25search2turn25search3

The main risks are clear. **Risk one** is premature endpointing, which makes the assistant feel rude; mitigate it with semantic turn detection, adjustable endpointing delays, and editable interim transcripts. **Risk two** is sluggish first audio; mitigate it with streaming TTS, smaller TTFT-optimized serving, and preemptive generation. **Risk three** is interruption chaos; mitigate it with adaptive interruption logic and aggressive cancellation propagation. **Risk four** is infrastructure overspend from chasing concurrency blindly; mitigate it with latency-aware batching and real profiling rather than raw autoscaling. citeturn23view0turn23view1turn23view2turn16view0turn18view0turn6search3turn6search11

The open questions that should be answered before buildout are operational rather than theoretical. Which voices and languages matter first. Whether transcripts must be durable, searchable, or compliance-scoped. Whether the product must support telephony or only browser and mobile sessions. Whether the LLM must remain self-hosted for policy or privacy reasons. Whether the team can tolerate managed speech dependencies to hit latency and quality targets sooner. And whether the design goal is merely “usable voice” or truly premium, interruptible, real-time conversation. The right architecture is materially different depending on those answers.

# ADR 0006 — Roles & Permissions (RBAC)

**Status:** Accepted (Phases 1–3 implemented)
**Date:** 2026-06-08
**Depends on:** ADR 0002 (Users), ADR 0003 (Canonical identity), ADR 0004 (Org invitations)
**Owner of orgs/members/roles/scopes:** `src/host/accessControlService.ts` +
`src/routes/accessControl.ts` (NOT a new feature — the single owner).
**Authored with:** the dual-track `/architect` skill (boundaries + wire/capability honesty).

> **No new OpenWOP RFC.** Roles → scopes is **RFC 0049** (`Draft`), whose
> `capabilities.authorization` block + `authorization.decided` event + the
> `/v1/host/sample/authorization/decide` seam contract **already shipped in the
> spec corpus on `main`** (`capabilities.schema.json §authorization`,
> `host-sample-test-seams.md`). Phase 3 wires the host onto that existing
> surface — advertising `capabilities.authorization` is a wire claim gated on
> honest enforcement, not a schema change, so no new RFC is required.

---

## Context (boundaries audit first — the check the old skill lacked)

Before proposing anything, the audit (mandatory per the architect skill):
`accessControl` **already** models orgs / teams / members / **groups** / **roles**,
with roles mapped to the **RFC 0049 scope vocabulary** + `host:` management
scopes, resolved **fail-closed** from a member's explicit `roles[]` (never from
org-chart position — a SECURITY invariant). So RBAC is **not new** — accessControl
is the owner, and ADR 0006 **completes** it. (This is the same surface ADR 0004
was reconciled onto; building a parallel RBAC would repeat that mistake.)

Two gaps accessControl's own header names:

1. **Authority is `tenant == principal, implicitly owner`.** With no act-as
   member context, the caller is treated as the tenant owner. The header: *"when
   multi-principal tenants are real, replace it with an explicit owner member
   seeded at org creation."* Until then, two users in one tenant are BOTH
   implicitly owner — no per-user authority.
2. **Enforcement is management-only.** `requireScope` gates org/team/member CRUD,
   but **not** the protocol runs/artifacts paths — so the host deliberately does
   **NOT** advertise `capabilities.authorization` (advertising RFC 0049 it doesn't
   enforce on the wire would be a *false authorization-oracle*).

ADR 0003 (stable `User.userId` subject) and ADR 0004 (invitations that bind a
member to `User.userId`) supply exactly what's needed to close both.

## Decision

**The RBAC subject is `User.userId`; authority is membership-derived; the host
advertises `capabilities.authorization` ONLY once it honestly enforces RFC 0049
on the protocol surface.** Delivered in three phases.

### Phase 1 — Explicit, userId-bound org ownership  ✅ (this ADR)

Creating an org through `POST /v1/host/sample/orgs` now **seeds an explicit owner
`OrgMember` bound to the creating `User.userId`** (ADR 0003: `req.userId`, falling
back to the principal, then the tenant for the single-principal demo).
`createOrg` gained an opt-in `ownerSubject`; the route passes it.

This is **additive and back-compat**: `resolveEffectiveAccess`'s implicit
tenant-owner is unchanged (existing `access-control.test.ts` still passes), but
every route-created org now has an explicit owner member, so
`resolveEffectiveAccess(tenant, { subject: userId })` resolves the creator to
`owner` — the foundation for per-user authority. Composes with ADR 0004: the
owner and every invited member are now explicit, `userId`-bound members.

### Phase 2 — Membership-derived authority  ✅

`requireScope` and `/access/effective` now resolve the **caller's own subject**
(`callerSubject(req)` = `req.userId` ▸ principal), scoped to the org being acted
on (`req.params.orgId` → `resolveEffectiveAccess({ subject, orgId })`). An
authenticated **non-member resolves to ZERO scopes** — the implicit tenant-owner
shortcut is gone (the multi-principal hazard is closed: two users in one tenant
get distinct, membership-derived authority). The `x-openwop-act-as` demo seam
still overrides for "view as member".

`resolveEffectiveAccess` gained `orgId` so a subject that is a member of several
orgs resolves per-`(subject, org)`, not "first match in the tenant".

**Bootstrap (chicken-and-egg):** `POST /orgs` (org *creation*) can't require
membership — there's no org yet — so it gates on `requireAuthenticated` (any
signed-in principal) and the creator is seeded as the explicit owner (Phase 1).
Authority on *existing* orgs is membership-derived.

`access-control.test.ts` updated: the `basis: 'tenant-owner'` introspection
expectation became `basis: 'member'` (creator = owner) + a new non-member →
`basis: 'none'` fail-closed assertion. All management/enforcement tests passed
unchanged (the seeded owner makes the caller resolve to owner per-org).

### Phase 3 — Protocol-surface enforcement + honest capability  ✅

`requireProtocolScope` (new `src/host/protocolAuthorization.ts`) now gates the
protocol runs/artifacts routes — `POST /v1/runs` + fork (`runs:create`),
`GET /v1/runs` + `GET /v1/runs/:runId` (`runs:read`), cancel (`runs:cancel`),
the artifacts stub (`artifacts:read`) — resolving the caller's
membership-derived RFC 0049 scopes and failing closed with the canonical
`forbidden` envelope. The `POST /v1/host/sample/authorization/decide` seam
(RFC 0049 §C) returns `{ allowed }` fail-closed: an absent/unseeded principal
(`basis: 'none'`, zero scopes) is denied; the host never default-allows.

**The honesty gate is the whole point.** All of this — enforcement, the seam,
and the `capabilities.authorization.supported: true` advertisement — is gated on
`OPENWOP_AUTHORIZATION_ENFORCEMENT`. OFF (default): `requireProtocolScope` is a
no-op, the seam 404s, discovery advertises `authorization.supported: false`, and
every existing protocol caller (the conformance harness; Bearer principals with
a tenant allow-list but no accessControl membership) is unaffected — the full
suite stays green. ON: enforcement is real, so the capability is advertised and
the conformance leg `authorization-fail-closed.test.ts` runs **non-vacuously**.
The host advertises RFC 0049 **iff** it honors it — never a false
authorization-oracle. `test/authorization-fail-closed.test.ts` (route harness)
proves both postures over HTTP, including the non-vacuous allow ∧ deny pair.

No new RFC — the `capabilities.authorization` block, the `authorization.decided`
event, and the seam contract already shipped in the spec corpus (RFC 0049,
`Draft`). It IS a wire-visible capability flip, so it is additive,
forward-compatible, and conformance-gated.

> **Phase 3 follow-up (code-review corrections, same date).** A high-effort
> review of the Phase 3 merge surfaced gaps that were fixed in
> `feat/rbac-phase3-followup` (the original Phase 3 rationale above stands; these
> are completeness/hardening corrections):
> - **Enforcement completeness.** The first cut gated only the "main" routes.
>   `POST /v1/runs:bulk-cancel` (a mutation), `DELETE /v1/runs/{id}`, and the
>   read routes `…:diff` / `…/ancestry` / `…/events/poll` / `…/debug-bundle` were
>   ungated — closed now, with an explicit route→scope inventory at the top of
>   `registerRunRoutes` as the audit surface so a new route can't silently ship
>   unprotected. (Annotations stay ungated by design — RFC 0056 feedback surface,
>   not RFC 0049.)
> - **Multi-org authority.** Protocol authority now resolves the **union** of a
>   subject's scopes across all its org memberships (`resolveSubjectScopesUnion`),
>   not the org-scoped first-match of `resolveEffectiveAccess` — a subject that is
>   `viewer` in one org and `editor` in another no longer resolves
>   non-deterministically.
> - **Fail-closed on resolver error.** A resolver throw now resolves to zero
>   scopes (logged) rather than a 500, honoring the advertised `failClosed: true`.
> - **Decision-seam disclosure.** The seam returns only `{ allowed }` (its
>   documented contract); it no longer echoes the principal's scope list / basis
>   (an enumeration surface).
> - **De-duplication.** `callerSubject`/`tenantOf` now live in one
>   `host/requestSubject.ts`, imported by both the management and protocol
>   surfaces so they cannot diverge.

## Architectural constraints honored (dual-track architect)

- **Boundaries / single source of truth:** accessControl owns orgs/members/roles;
  ADR 0006 extends it — no parallel RBAC system (the ADR-0004 lesson).
- **Capability honesty (wire):** `capabilities.authorization` advertised only when
  enforced on the protocol surface + conformance-green (Phase 3 gate).
- **Identity (ADR 0003):** the subject is `User.userId`, stable across logins /
  auth methods — not an auth-method principal or the tenant.
- **Fail-closed (RFC 0049):** no member / no role ⇒ zero scopes; the decision seam
  denies an unknown principal.

## Alternatives considered

1. **A new `rbac` feature package.** Rejected — duplicates accessControl (the
   exact collision ADR 0004 had to unwind). RBAC extends the owner.
2. **Flip to strict authority immediately (Phase 2 now).** Rejected as one step —
   it's a behavior change that rewrites `access-control.test.ts`; sequencing it
   separately keeps each PR's blast radius and the harness honest.
3. **Advertise `capabilities.authorization` now.** Rejected — a false oracle until
   the protocol surface is enforced (Phase 3).

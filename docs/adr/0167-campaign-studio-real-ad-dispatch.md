## Context

ADR 0166 shipped the honest stopgap for `ad_variants`: a **document handoff** the user pastes into the ad platform, because openwop-app had no outbound platform dispatch. It named the real thing — pushing a generated ad to a live Meta/Google/TikTok campaign — as "a large, RFC-gated effort." **That RFC-gated assumption was wrong**, and an intake corrected it:

- **The wire is already specified (Accepted).** RFC 0045: an *action* is a normal side-effecting node that makes an authenticated outbound call to create a remote resource — *"it does not introduce a new execution kind."* RFC 0046/0047: `connectionId`→OAuth-token resolution is host-side, sandbox-only, **never on the wire**. RFC 0079: credential-bearing **egress** is audience-bound with `egress.decided` events. `idempotency.md` Layer 2 + `replay.md` make a remote create **replay-safe** (`invocationId = sha256(runId:nodeId:attempt:providerKey)` + recorded-fact events).
- **openwop-app already has the host arm.** The Connections broker does OAuth `authorization_code` + refresh (`features/connections/{oauthFlow,refreshDaemon,serviceAccountJwt}.ts`); `host/brokeredEgress.ts` exposes `brokeredPost`/`brokeredFetch` (resolves a Connection's secret into the `Authorization` header and posts); **`slackAdapter`/`smsAdapter`/`emailAdapter` already do credentialed outbound dispatch through the broker** — the exact pattern. The host advertises `connections.packsSupported`.
- **Reference packs exist.** `vendor.myndhyve.ads-publish-{meta,google,tiktok}` already do the Meta v21 / Google v18 / TikTok v1.3 pipelines (created-paused, cascade rollback) on MyndHyve's `workflow-runtime`, passing RFC 0045/0046/0047 conformance — proving the mechanism.

What's **missing** in openwop-app is small and concrete: the ad connection packs (`google-ads`/`meta-ads`/`linkedin-ads`) declare `provider`/`auth`/`reach` but **no RFC 0045 `actions`**; there's no ads adapter composing the broker; and `publish-ad-variants` only writes a document. This ADR closes that — **no new wire, no new RFC.**

## Decision

**Add a host `ads-dispatch` adapter (the `slackAdapter` precedent) that composes `brokeredPost`/`brokeredFetch` through a configured ad-platform Connection, and have `publish-ad-variants` invoke it when a Connection is present — else fall back to the ADR 0166 document handoff.** Honest degradation: dispatch only when a real connection is configured; otherwise the working document path.

1. **`host/adsAdapter.ts`** — `makeAdsAdapter(deps: BrokeredEgressDeps)` exposing `publishAd({ platform, connectionId, campaign, adSet, ad, creative })` → runs the platform's create pipeline via `brokeredPost` (campaign → ad-set/group → ad), **created `PAUSED`**, returns `{ platformCampaignId, platformAdSetId, platformAdId, reviewStatus }`. Per-platform request mappers (Meta graph v21, Google ads v18, TikTok business v1.3). The broker owns auth; the adapter owns the per-platform request/response shape — exactly as `slackAdapter` owns the Slack body shape. **Adapter invariants (load-bearing):**
   - **Pin the egress host.** `brokeredPost` does **not** pin the destination to the provider's `apiHosts` (only `brokeredFetch` does); so the adapter MUST build the URL from a **hardcoded host constant** (`graph.facebook.com` / `googleads.googleapis.com` / `business-api.tiktok.com`) and **never** derive it from brief/agent-controlled input — else a token-bearing request could be aimed off-provider.
   - **Stamp provenance explicitly.** Call `stampConnectionUse(...)` per successful create (the `slackAdapter` precedent) so the RFC 0079 audience-binding record lands in `run.metadata.connectionUse[]` — it is **not** automatic.
   - **created-PAUSED is a literal in the mapper**, never an overridable input.
   - **TikTok uses an `Access-Token` header, not `Bearer`** — `brokeredPost`'s `authScheme` only does bearer/basic, so the adapter adds a custom-header path for TikTok.
   - **Google needs a second header** (`developer-token`) on every call. **Correction (Phase 2, per `/architect`):** this is NOT a per-user Connection secret — the developer-token is an **app/manager-account-level** credential (one per integration), so it lives host-side as **operator config** (`OPENWOP_GOOGLE_ADS_DEVELOPER_TOKEN`, read through a `resolveDeveloperToken(tenantId)` seam that leaves the BYOK-per-tenant door open), mirroring `OPENWOP_OAUTH_*_CLIENT_SECRET`. The broker stays the sole authority for the per-user OAuth token; the developer-token rides a new `brokeredPost` `extraHeaders` param (which strips any `authorization` override).
2. **RFC 0045 `actions[]` on the ad connection packs** — `examples/connection-packs/{meta-ads,google-ads}/pack.json` gain `actions` (`create-campaign`, `create-adset`, `create-ad`) binding to the adapter, so the capability is portably declared (RFC 0045 §B: each `actions[].typeId` resolves to a real node).
3. **`publish-ad-variants` (ADR 0166) gains a dispatch branch** — if the brief/inputs resolve an ad-platform `connectionId`, invoke `ctx`'s ads adapter to publish (paused); **on no connection / capability-missing → the existing document handoff** (unchanged). The node returns either `{ campaign: { platformCampaignId, reviewStatus } }` or `{ document }`.
4. **Replay-safe via an adapter-owned idempotency map** (NOT "by construction"). The host's Layer-2 invocation log (`executor/invocationLog.ts`) is keyed on `(runId, nodeId, attempt)` and is consumed only by `aiProvidersHost` — `brokeredPost`/the existing adapters do **not** consult it, and a `:fork` mints a **new runId**, so it cannot dedup across a fork. The local publish nodes (ADR 0162/0166) are fork-safe by deriving a deterministic *local* id from stable inputs and short-circuiting — but an ad platform **assigns** the id, so that trick is unavailable. Therefore `adsAdapter` MUST keep its **own durable idempotency-key → platformCampaignId map**, keyed on **fork-stable business inputs** (`briefId:platform:adObjectHash`, never `runId`), **consulted before** the `brokeredPost` create: a same-run retry, a sweeper re-dispatch, or a `:fork` all return the recorded platform id — **no duplicate paid campaign**. The platform ids surface in the node `outputs` (the slack/sms/email precedent emits no custom run-event — it returns outputs + stamps provenance); they are content-free (no token, no creative bytes).

### Key design decisions

| Decision | Choice | Rationale |
|---|---|---|
| New RFC? | **No** — rides Accepted RFC 0045/0046/0047/0079/0095 | The intake proved the surface exists; a new RFC re-specs 0045. Host work per the "rides an Accepted RFC ⇒ no RFC" rule. |
| Mechanism | A broker-composing **adapter** (slack/sms/email precedent), not a bespoke HTTP client | Reuses OAuth resolution, refresh, egress audience-binding, secret redaction — no second credential path. |
| Created state | **PAUSED is a hard invariant** — the adapter never creates an active/spending ad | No auto-spend; a human activates on-platform. The single most important safety norm. |
| Human gate | A `core.approvalGate` predecessor MUST sit on the dispatch branch — `publish-ad-variants` runs only after explicit human approval | AI-generated ad copy must pass a person before a paid object is created. **Structural, not asserted:** the workflow that runs `publish-ad-variants` (the orchestration dispatch step) MUST gate it behind an approval node — `created-PAUSED` is the backstop, not the gate. |
| Degradation | No connection ⇒ the ADR 0166 **document handoff** | Honest: the working path always exists; dispatch is the upgrade when a real connection is present. |
| Idempotency / replay | An **adapter-owned** durable idem-key→platformId map keyed on fork-stable inputs (`briefId:platform:adObjectHash`) | The Layer-2 `invocationId` log is `runId`-keyed (fork-unsafe) **and** not consulted by the egress path; the deterministic-local-id trick (ADR 0162/0166) can't apply to a platform-assigned id. A fork/retry MUST NOT create a second campaign (`replay.md`). |
| Credential safety | Token resolved in-broker only (`brokeredPost` → `resolveConnectionCredential`); never passed by the node, never in inputs/events/debug/replay | Inherits `credential-payload-redaction` (RFC 0046). Audience-binding (RFC 0079) is **explicit** — the adapter MUST call `stampConnectionUse` + pin the host; it is not free. |
| Run events | **None custom** — ids ride node `outputs` + `connectionUse` provenance (the slack/sms/email precedent) | Avoids inventing an `ad.*` event family; nothing that even looks like a wire addition. |

### Non-goals

- **Organic social posting** (LinkedIn/X/IG feed posts) — no platform API for it exists in the source (MyndHyve is ad-only); `social_posts` stays the ADR 0166 document handoff.
- **Budget optimization, scheduling/dayparting, A/B rotation, performance-driven pausing** — later (ADR 0159 connectors own inbound metrics; activation timing is a follow-on).
- **Auto-activation** — out of scope by safety design (created-paused MUST).
- **A new capability flag / event family on the wire** — the intake found `egress.decided` (RFC 0079) + recorded-fact events already cover it; no normative additions.

> **Known limitation (all three phases):** the per-platform request **bodies are minimal — the dispatch SEAM, not production-complete payloads.** Live Meta/Google/TikTok create calls require more fields (e.g. Meta creative+page, Google RSA already padded but needs a real `finalUrls`/landing flow, TikTok adgroup `placements`/`schedule`/`optimization_goal` + ad `identity_id`/media). They will 400 against the live APIs until fleshed out — bounded-safe (a rejection fails closed → no spend; everything is created PAUSED regardless). Completing the payloads (incl. asset/media upload) is the follow-on past the seam.

## Alternatives considered

1. **Author RFC 0120 (a new outbound-dispatch capability).** Rejected — the intake proved RFC 0045/0046/0047/0079 already specify it + reference packs are live; a new RFC re-specs 0045 (fork).
2. **Keep only the ADR 0166 document handoff.** Rejected as the ceiling — it's the honest *fallback*, but a configured connection should publish for real (the whole point of Campaign Studio's last mile).
3. **A bespoke per-platform HTTP client (port MyndHyve's clients verbatim).** Rejected — bypasses the broker's OAuth/refresh/egress-audience/redaction guarantees; the adapter-over-broker pattern (slack/sms/email) is the sanctioned seam.
4. **Auto-activate created ads.** Rejected — created-paused is a safety MUST; no auto-spend.

## Consequences

- A configured Meta/Google/TikTok connection turns an approved `ad_variants` draft into a real **paused** campaign in one chat turn; no connection → the ADR 0166 document, unchanged. The five-channel last mile is now real where a connection exists.
- Establishes the **ads-dispatch adapter** as the outbound-platform seam (the same shape future paid channels reuse).
- Corrects ADR 0166's "RFC-gated" deferral: outbound dispatch was always host work riding Accepted RFCs.

## Operator setup — enabling real dispatch

Real dispatch is **opt-in and off until configured**; with nothing set up, `publish-ad-variants`
produces the ADR 0166 document handoff (no platform calls). To enable it:

1. **Connect the platform (per acting user).** The user authorizes a Connection for the
   provider — `meta-ads`, `google-ads`, and/or `tiktok-ads` — through the existing Connections
   broker (OAuth). The operator supplies the OAuth **client id/secret host-side** (ADR 0024 §7 /
   `OPENWOP_OAUTH_*`), never in the connection pack.
2. **Google only — set the developer-token.** `OPENWOP_GOOGLE_ADS_DEVELOPER_TOKEN=<token>` (an
   operator/manager-account-level credential, not per-user). **Without it, Google dispatch
   fails closed** → the document handoff (it never sends a blank-header call).
3. **Target a real account to dispatch.** The agent/caller passes the node `adAccountId` (the
   public Meta `act_<id>` / Google customer id / TikTok advertiser id) and `platform`
   (`meta` | `google` | `tiktok`, default `meta`); for Google also a `landingUrl`. No
   `adAccountId` ⇒ document handoff (the safe default).
4. **Safety — campaigns are created PAUSED.** A dispatched campaign **never spends**; a human
   reviews and **activates it in the ad platform UI**. The Campaign Strategist agent additionally
   gates dispatch behind explicit chat approval (the agent-driven HITL).

> **`OPENWOP_{META,GOOGLE_ADS,TIKTOK_ADS}_API_BASE` are TEST-ONLY** overrides (point the adapter
> at a mock); **never set them in production** — the adapter pins the real platform hosts by default.

**Known limitations (see Non-goals + the Phase-3 note):** the per-platform request **bodies are
the seam, not production-complete** (they will 400 against the live APIs until fleshed out — but
fail closed, no spend); **media/asset upload** is not yet wired; and the Meta cascade-delete
rollback is best-effort PAUSED-safe until **RFC 0120** (connection-pack `apiHosts`) lands and the
ad packs declare their API hosts (then `brokeredFetch` DELETE works for real).
